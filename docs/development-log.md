# Development Log

## v0.2 Snapshot

This snapshot records the current CoreS3 prototype state before publishing to GitHub as `xhs_ai_builder` v0.2.

## Implemented Flow

1. Basic CoreS3 flashing and official firmware recovery path.
2. Local Arduino demos for display, touch, Wi-Fi scan, and camera capture.
3. Touch photo trigger.
4. Camera feature extraction into local image traits.
5. Subject detection and 8-class trained feature-model recognition before pet generation.
6. Capture failure screen when no clear subject or low confidence is detected.
7. Five-element pet generation with visible species and gene variation.
8. Backpack capture, selection, release, and release confirmation.
9. Pet growth fields and simple level/stat progression.
10. Scene, pet, battle, and startup sound system.
11. Two-board battle flow.
12. Communication replacement from ESP-NOW to Wi-Fi AP + UDP for easier pairing and debugging.
13. Input now uses touch and reserved external action interfaces.
14. Public shared types were split into `pet_model.h`, `vision_types.h`, `ui_types.h`, and `battle_protocol.h`.
15. Productization planning was added for App, desktop MVP, cloud hints, SD storage, datasets, community, and interface boundaries.
16. Player-facing MATCH/BATTLE documentation now uses role-neutral wording; HOST/CLIENT details remain developer diagnostics only.
17. v0.2 release notes were added in `docs/release-v0.2.md`.

## Current Operation Notes

- `PHOTO` captures a frame, recognizes a subject, then creates a wild pet only on success.
- Failed recognition shows CAPTURE FAILED with presence score, confidence, timing, and heap data.
- The first trained model is generated by `scripts/train_vision_feature_model.py` and embedded through `vision_model_data.h`.
- The recognizer remains a lightweight limited-class local model, not a general vision large model.
- `BAG` opens stored pets.
- Backpack persistence uses ESP32 `Preferences`; no SD card is required.
- `MATCH` starts peer discovery and battle.
- Both boards run identical firmware.
- The matching screen exposes role and packet counters to diagnose communication:
  - `HOST` or `CLIENT`
  - UDP port
  - `TX`
  - `RX`
  - `F`

## Reserved Work

- Voice recognition is not restored.
- App, cloud recognition, SD storage, LLM text generation, and community features are productization directions, not current runtime dependencies.
- TFLite Micro or Edge Impulse image model to replace the current trained feature classifier.
- SD card storage for snapshots, history, and audio assets.
- Wi-Fi API integration for remote pet hints.
- More robust physical button input through the existing external action interfaces.
- Better memory budgeting and optional asset loading from SD card.
