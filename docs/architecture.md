# Architecture

## Main Sketch

`arduino_demos/04_camera_pet_battle/04_camera_pet_battle.ino` contains the v0.1 system:

- Camera capture, subject detection, 64x64 vision preprocessing, and 8-class heuristic recognition.
- Five-element pet generation.
- Pet rendering on the CoreS3 screen.
- Backpack persistence with `Preferences`.
- Growth, XP, win rate, and battle stat calculation.
- Wi-Fi AP + UDP battle transport.
- Scene and pet sound effects.
- Touch and reserved external action inputs.

No voice recognition is implemented. Audio is limited to local sound effects and startup audio, and the sound system must respect the firmware mute setting.

## Public Type Boundaries

Shared types live in public headers and must not be redefined in modules:

- `pet_model.h`: `ElementType`, `PetGenes`, `SavedPet`, `BackpackStorage`, `kMaxBackpackPets`, `kBagMagic`, and `kBagVersion`.
- `vision_types.h`: `ObjectClass`, `ImageTraits`, `RecognitionResult`, `SubjectPresence`, and `PetHint`.
- `ui_types.h`: `ScreenMode`, `UiAction`, and `ButtonSlot`.
- `battle_protocol.h`: `BattleLinkRole`, `BattlePetPacket`, `kBattleMagic`, and `kBattleVersion`.

Public enum values, struct fields, storage constants, and the packed UDP packet layout are compatibility-sensitive. If a future App, SD, cloud, or training module needs a new public field or protocol command, it should first produce an interface change request instead of changing these headers locally.

## Pet Generation

The generated pet is represented by `PetGenes`:

- `element`
- `species`
- `mood`
- `bodyScale`
- `eyeStyle`
- `hornStyle`
- `tailStyle`
- `auraPattern`
- `patternDensity`
- `accentColor`
- `seed`

The main local inputs are:

- Average image color.
- Brightness.
- Saturation.
- Contrast.
- Recognition class and confidence.
- Subject presence score.
- Time bucket.
- Shot count.

The design goal is to only generate pets when a clear object is recognized. Class ID controls the element/species bias, while image traits and shot count only vary the appearance within that recognized class.

## Local Recognition

The current recognizer is an offline first-stage implementation:

- It rejects frames that are too dark, too bright, too flat, or lack center/edge separation.
- It down-samples the camera frame to a 64x64 int8 RGB input buffer.
- It runs a tiny trained softmax feature model generated by `scripts/train_vision_feature_model.py`.
- It classifies recognized subjects into 8 local classes: plant/leaf, food/fruit, paper/book, electronics/screen, metal/key/coin, fabric/cloth, cup/bottle/water, and toy/figure.
- It falls back to heuristic classification if the trained model is too uncertain.
- It keeps `vision_model_data.h` as the model data entry point; this can later be replaced by a TFLite Micro or Edge Impulse int8 model.
- It displays and logs preprocessing time, classification time, presence score, confidence, and free heap.

This recognizer is a lightweight limited-class local model. It is not a general-purpose vision model and does not run YOLO, CLIP, or a visual large model on CoreS3.

## Extension Interfaces

The sketch keeps these interfaces for later upgrades:

- `RecognitionResult recognize_object_local(const ImageTraits& traits)`
- `bool fetch_remote_pet_hint(const RecognitionResult& input, PetHint* output)`
- `bool save_pet_snapshot(const PetGenes& genes, const ImageTraits& traits)`
- `PetGenes merge_generation_inputs(ImageTraits traits, RecognitionResult recog, PetHint remoteHint)`

Remote and SD paths remain local/no-op. Later App, network API, SD card history, or trained model recognition work should be integrated behind these interfaces so the drawing and gameplay layers do not need major rewrites. These extensions must remain optional, with local capture, backpack, and battle flows continuing to work offline.

## Battle Transport

The v0.1 battle module replaced ESP-NOW with Wi-Fi SoftAP + UDP:

- Every board starts a SoftAP with SSID `M5PET-xxxxxxxx`.
- Every board listens on UDP port `42105`.
- During MATCH, boards scan for peer APs.
- The larger device ID joins the smaller device ID AP.
- The host learns the client IP from the first UDP packet.
- Both boards exchange `BattlePetPacket` structures.

This design is easier to debug than ESP-NOW because the screen can show role, connection state, IP, TX count, RX count, and send failures.

The player-facing battle flow is:

1. `MATCH`: exchange active pet packets.
2. `BATTLE` clash: show a short local battle-process screen.
3. `BATTLE` result: show both stats, scores, result, XP gain, and rematch hint.

Repeated battles against the same recent peer add a small local friendship XP
bonus. The friendship state is RAM-only in v0.1, so no public storage or UDP
layout change is required.

## Persistent Data

Backpack data is stored with ESP32 `Preferences`.

Each saved pet includes:

- `PetGenes`
- Level and stage.
- XP.
- Battle count and win count.
- Capture timestamp.
- Last growth timestamp.

No SD card is required in v0.1.

## Productization Extensions

Productization planning is tracked in `docs/app-cloud-roadmap.md` and `docs/module-tasks.md`.

Near-term App or desktop tooling should prefer a separate management/debug protocol over changing the battle packet:

- Phase 1: USB serial for status, backpack summaries, pet details, recognition results, battle status, logs, and sample export.
- Phase 2: Wi-Fi HTTP/UDP for local App access. UDP remains the battle transport.
- Phase 3: BLE or LAN discovery after resource and stability evaluation.

Cloud recognition, LLM-generated flavor text, SD card storage, datasets, and community features are future optional layers. They must not introduce hard-coded API keys, mandatory uploads, mandatory network access, or a dependency on SD card availability.
