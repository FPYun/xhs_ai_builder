# Mobile App HTTP API

This document records the implemented local HTTP management API for the Android
Chrome / iPhone Safari `/app` phone page and the `M5PetCompanion-iOS` App. It is
separate from the Wi-Fi AP + UDP battle packet transport and does not modify
`BattlePetPacket`.

## Runtime Boundary

- The phone must manually connect to the CoreS3 `M5PET-xxxxxx` Wi-Fi AP.
- The API listens on the CoreS3 AP IP, for example `http://10.23.<subnet>.1`.
- The built-in phone page is available at `http://10.23.<subnet>.1/app` in
  Android Chrome or iPhone Safari.
- The API is local-only and does not require cloud, router, SD card, BLE, or an
  external network.
- JSON is generated by the firmware directly; no ArduinoJson or third-party
  dependency is used.
- The UDP battle link remains separate and continues to use `BattlePetPacket`.

## App Connection Flow

1. User connects the phone Wi-Fi to the CoreS3 `M5PET-xxxxxx` AP.
2. The App reads the current Wi-Fi gateway IP and uses `http://<gateway-ip>` as
   the default base URL.
3. A phone browser can open `/app`, or a native App can request
   `GET /api/v1/status`.
4. A valid connection must return `ok:true`, `ssid`, `deviceId`, and
   `httpBaseUrl`.
5. The App polls `GET /api/v1/status` every 1-2 seconds and refreshes
   immediately after each successful action POST.

If the first status request fails, keep the user on the connection screen and
ask them to confirm they are connected to the `M5PET-xxxxxx` AP. Do not fall
back to BLE, cloud, or ESP-NOW in v0.1.

## Endpoints

| Method | Path | Purpose |
| --- | --- | --- |
| GET | `/app` | Built-in mobile browser page for status, backpack, capture, battle, and logs. |
| GET | `/api/v1/status` | Device, screen, current pet, recognition, and battle summary. |
| GET | `/api/v1/backpack` | Compact pet cards for the backpack list. |
| GET | `/api/v1/encyclopedia` | Read-only 45-template pet encyclopedia derived from base element/species/variant names and current backpack ownership. |
| GET | `/api/v1/pets?index=N` | One pet detail record. |
| POST | `/api/v1/action?type=photo` | Trigger the existing PHOTO flow. |
| POST | `/api/v1/action?type=bag` | Open BAG. |
| POST | `/api/v1/action?type=match` | Enter MATCH. |
| POST | `/api/v1/action?type=idle` | Return to IDLE. |
| POST | `/api/v1/action?type=prev` | Move to previous backpack pet. |
| POST | `/api/v1/action?type=next` | Move to next backpack pet. |
| POST | `/api/v1/action?type=select` | Select current backpack pet. |
| POST | `/api/v1/action?type=capture` | Capture the current wild pet when WILD is visible. |
| POST | `/api/v1/action?type=release` | Release wild pet, or open stored-pet release flow. |
| POST | `/api/v1/action?type=confirm_release` | Confirm stored-pet release only while RELEASE_CONFIRM is active. |
| POST | `/api/v1/action?type=cancel` | Cancel release confirmation or return to IDLE. |
| POST | `/api/v1/action?type=friend` | Add the currently seen opponent, or the most recent opponent, to the local persisted friend list. |
| POST | `/api/v1/action?type=mute` | Mute device sound for the current runtime. |
| POST | `/api/v1/action?type=unmute` | Enable device sound for the current runtime. |
| POST | `/api/v1/action?type=toggle_mute` | Toggle device sound for the current runtime. |
| GET | `/api/v1/recognition/last` | Last `ImageTraits` and `RecognitionResult` display summary. |
| GET | `/api/v1/battle` | Role-neutral battle phase and result summary. |
| GET | `/api/v1/battle/replays` | RAM-only recent battle replay records, newest first. |
| GET | `/api/v1/logs` | RAM-only App event log, limited to the most recent 20 events. |
| GET | `/api/v1/storage` | SD card type, capacity, used/free bytes, capped file list, and expected audio asset presence. |
| GET/POST | `/api/v1/sampling?enabled=0|1&label=...&scene=...` | Read or update RAM-only sampling mode for SD dataset capture labels. |
| GET | `/api/v1/capture/quality` | Probe one camera frame for pre-shot quality meter data without recognizing or generating a pet. |
| GET | `/api/v1/samples` | SD `/samples/manifest.csv` summary with class counts, scene counts, and recent sample rows. |
| GET | `/api/v1/samples/manifest` | Download the SD sample manifest as `manifest.csv`. |
| GET | `/api/v1/samples/file?path=/samples/...` | Download one allowed local sample artifact under `/samples/`, such as a PPM thumbnail or CSV file. |

## JSON Notes

- Battle phase values are role-neutral: `finding`, `connected`, `ready`,
  `clash`, or `result`.
- Battle status fields stay player-facing: `pairingStep`,
  `pairingStepLabel`, `waitingSec`, and `syncAgeSec` describe progress without
  exposing transport counters such as TX/RX/failures.
- During BATTLE CLASH, `clashRound`, `clashRoundLabel`, and `clashRoundDiff`
  expose the same current-round state shown by the device's 力/克/心 track and
  center impact visuals.
- Battle responses include a RAM-only `replays` array with up to 6 local-view
  records: `battleId`, result, local display skill names, scores, 力/克/心
  diffs, opponent base species, reward XP, friend bonus, and growth flags.
  `/api/v1/battle/replays` exposes the same array without the live battle
  summary. This is App display/debug state only; it is not stored in
  `BackpackStorage` and is not sent over UDP.
- `sdCardPresent` is a read-only boot-time SD mount check. `/api/v1/storage`
  adds SD card type, total/used/free bytes, a capped recursive file list,
  whether each expected `/audio/.../*.raw` asset exists, and whether optional
  `/skins/manifest.csv` and `/actions/manifest.csv` addon manifests exist. The
  current firmware does not write gameplay data on the SD card. When present,
  `/skins/palettes/default.csv` and `/skins/palettes/<element>.csv` may override
  local pet drawing `body` and `accent` colors; `/actions/*.csv` may tune local
  pet drawing `bob`, `sparkle`, and `tilt` by screen. Bad or missing files fall
  back to built-in colors and static drawing.
- `/api/v1/samples` is read-only and summarizes local capture logs stored on
  the SD card. It reports counts by current 8 object classes plus `negative`,
  counts by scene labels such as `white_wall`, `white_paper`, `desktop`,
  `glare`, `dark`, `bright`, and recent manifest rows.
- `/api/v1/sampling` is RAM-only. When enabled, SD sample rows use the operator
  label and scene selected in `/app`; recognition and capture gates still use
  the real local recognition result. The manifest keeps `labelSource` and
  `autoLabel` so manual samples remain auditable. New rows also include
  `distanceHint`, a non-protocol training aid such as `near`, `too_far`,
  `flat_background`, `visual_near`, or `distance_unknown`.
- USB serial `SAMPLE ...` commands update the same RAM-only sampling state as
  `/api/v1/sampling`. This is for bench capture sessions without a browser and
  does not change recognition gates, storage structs, or UDP packets.
- `/api/v1/capture/quality` takes one lightweight camera probe and returns
  brightness, saturation, contrast, center delta, dark/bright ratios, subject
  score, proximity, scene, and quality hint. It does not call pet generation and
  does not write a sample row by itself.
- `/api/v1/status` embeds `recognition.burstCandidates` after a photo attempt.
  Each row is RAM-only debug data for one burst frame: quality, presence,
  confidence, best distance, margin, source, image traits, and failure reason.
  It is not written to `BackpackStorage` and is not sent over UDP.
- `/api/v1/samples/manifest` and `/api/v1/samples/file` are intended for
  offline dataset export. They do not upload images, do not require cloud
  services, and do not change `RecognitionResult`, `PetGenes`, `SavedPet`, or
  `BattlePetPacket`.
- `/app` inserts a local sample-debug panel on the capture page. The panel can
  select manual sample labels/scenes, poll the pre-shot quality meter, refresh
  sample counts, show recent sample rows, and download the manifest for PC-side
  retraining.
- Backpack pet records serialize existing `SavedPet` data without adding storage
  fields.
- `/api/v1/encyclopedia` returns all `5 x 3 x 3` local visual templates and
  marks entries owned when the current backpack contains a matching
  element/species/visual-variant combination. It is derived at request time and
  adds no persistent encyclopedia storage.
- Backpack responses include `cursorIndex` for the current device-side backpack
  cursor. Status responses also include `bagCursor`. These fields are App HTTP
  display state only and do not change `BackpackStorage`.
- Pet records include `growthGoal`, a player-facing next evolution/level XP
  target generated from existing `level` and `xp`.
- Pet records include a RAM-only `care` object derived from existing pet fields:
  `fullness`, `energy`, `affection`, `focus`, `mood`, and `hint`. These values
  are display helpers only and are not written to `SavedPet`.
- Friendship fields are local device status values saved in the independent
  `wuxingfr` Preferences namespace and do not change `SavedPet`;
  `friendshipGoal` mirrors the on-device 好友/密友 goal badge.
- `buttons.left`, `buttons.middle`, and `buttons.right` mirror the current
  on-device three-button labels for external button, serial, or App adapters.
- `buttonActions.left`, `buttonActions.middle`, and `buttonActions.right`
  provide the matching `/api/v1/action?type=...` values, so `/app` and native
  tools can execute the same state-machine actions as the physical three-button
  footer without duplicating screen logic.
- `/app` renders the same dynamic three-button footer on home, capture,
  backpack, and battle tabs. This keeps stored-pet release two-step: `release` opens
  RELEASE_CONFIRM, and `confirm_release` is accepted only after that screen is
  active.
- `/app` includes local browser-only button preferences: large button mode for
  phone operation and confirmation before release/confirm-release actions. These
  preferences are stored in browser `localStorage`; they do not change firmware
  storage or public protocol types.
- `/app` splits the phone experience into six bottom-tab pages: home, capture,
  backpack, battle, friends, and more. Friends is a first-level social page
  because recent opponents, bond stage, and rematch rewards are core interaction
  signals, not secondary settings.
- The bottom tab bar uses delegated `pointerup` and `click` handlers so Android
  Chrome and iPhone Safari can switch pages reliably while status refreshes are
  running. Refresh/busy state only disables action buttons inside `main`, not
  the tab bar.
- The home page focuses on device state, current-pet summary, recent activity,
  and a small next-action area. It does not expose low-level transport details.
- Backpack cards are shown as a one-pet mobile pager. Left/right swipes call the
  same local HTTP actions as the buttons (`next` / `prev`) and update the
  device-side backpack cursor.
- Backpack cards include firmware-derived pet stats plus a browser-generated SVG
  avatar image. The avatar varies by element, species, variant, stage, and stats;
  it is display only and does not add image storage to `SavedPet`.
- The more page groups secondary companion functions: local friends, local
  event history, device settings, and connection guidance.
- The friends section renders local persisted friend records, including recent
  peer, friendship score, label, battle count, and follow-up suggestions. It is
  inspired by buddy/friend game loops but does not add account, cloud, or
  account, cloud, or public pet storage fields.
- The settings section can mute or unmute device sound for the current runtime
  via local HTTP action. The current firmware default is still local/offline and
  no audio preference is written to public storage.
- The event list shows up to the latest 20 RAM events. The browser page can
  filter them by category, show a small count badge on each category tab, and
  open each item in `/app` to show time, category, level, message, and a short
  detail note.
- Extra browser-only convenience features, such as favorite marks and coaching
  tips, are local UI helpers inspired by collection/virtual-pet games. They do
  not create cloud, account, SD-card, or new firmware persistence requirements.
- State-specific actions are accepted only on their matching screens:
  `next`, `select`, and stored-pet `release` require BAG; `capture` and wild
  `release` require WILD; `confirm_release` requires RELEASE_CONFIRM.
- Rejected state-specific actions return a player-facing `error` message such
  as "请先打开背包", "请先拍照发现野生伙伴", or "请先进入放生确认".
- App actions reuse existing UI actions and screen flow.
- The `friend` action is App-only/external-control only. It updates the local
  persisted friend list and never changes `SavedPet`, `BackpackStorage`, or
  `BattlePetPacket`.
- App action responses return the same status JSON shape as `GET /api/v1/status`.
- The `/app` page is firmware-hosted static HTML/CSS/JS. It uses `fetch()` to
  call the same `/api/v1/*` endpoints, includes mobile connection/error
  feedback, and does not upload images or use cloud services.
- Error responses use `{ "ok": false, "error": "..." }`.

## Compatibility Rules

- Do not extend this API by changing `BattlePetPacket`.
- Do not add fields to public headers for App-only display needs.
- If the App later needs persistent friendship, skills, cloud IDs, or richer
  model metadata, submit an interface change request before modifying public
  storage or protocol types.
- BLE, LAN discovery, AP passwords, pairing codes, or session tokens require a
  separate interface review before firmware implementation.
